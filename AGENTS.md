# Project Overview

**Courier SuperApps** â€” a mobile-first courier management system with customer management, waybill tracking, route optimization, and admin monitoring. Built with Next.js 16 + Drizzle ORM + PostgreSQL (Supabase).

**Primary audience:** Indonesian couriers/logistics. Indonesian (id) locale is the primary target; English (en) is the fallback.

**Design language:** Google Material Design 3 (MD3) Expressive â€” large border radii, glassmorphism, spring-based animations, and tactile micro-interactions (`active:scale-90` on all touchable elements). 

**Roadmap** at `ROADMAP.md`. Current milestone: v0.1.x (foundation & polish). Next major update: v2.0 (The Desktop Era) with full-screen command center, AI route optimization, real-time chat, and Redis support.

# Database Schema (`lib/schema.ts`)

All tables use `varchar(7)` primary keys generated by nanoid (custom alphabet excluding similar chars: `0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz`). Relations are defined via Drizzle's `relations()` API.

| Table | Notes |
|---|---|
| `users` | `name`, `email` (unique), `password` (bcrypt hashed), `role` (default `"courier"`) |
| `passwordResetTokens` | `email`, `token` (unique), `expires` |
| `customers` | `name`, `phoneNumber`, `address`, `latitude`/`longitude` (text!), `housePictureUrl`, `notes`. M2M with clusters via `customerClusters` |
| `deliveries` | `waybillNumber` (unique), `customerId` (FK cascade), `status` (default `"Pending"`), `codAmount` (text!), `receiverName`, `proofOfDeliveryUrl` |
| `clusters` | `name`, `notes`. M2M with customers via `customerClusters` |
| `logs` | Activity audit trail: `userId`, `userName`, `action`, `details`, `targetId` |
| `errorLogs` | Error tracking: `errorName`, `errorMessage`, `stackTrace`, `pathname` |
| `accessLogs` | HTTP access log: `pathname`, `method`, `ipAddress`, `userAgent` |

**CRITICAL**: `latitude`/`longitude` and `codAmount` are stored as `text`, not numeric types. ID generation uses `lib/utils.ts` -> `generateId()` (nanoid 7-char).

# Routing & middleware (`proxy.ts`)

Middleware is defined in `proxy.ts` (NOT `middleware.ts`). Uses `withAuth` from `next-auth/middleware`.

- All non-API routes require auth (except `/login`, `/register`, `/forgot-password`, `/reset-password`)
- `/admin/*` routes require `role === "superadmin"`
- **Mobile-only enforcement**: desktop User-Agent redirects to `/not-mobile`
- Matcher: `"/((?!api|_next/static|_next/image|favicon.ico).*)"`

# Authentication

**NextAuth v4** with **CredentialsProvider** (no OAuth). JWT session strategy.

**Login fields**: `usernameOrEmail` + `password`. Auth via bcrypt compare against `users` table.

**Session user shape** (extended via callbacks):
```ts
{ id: string, name: string, email: string, role: "courier" | "superadmin" }
```

**API route auth pattern**: Use `getToken` from `next-auth/jwt`:
```ts
import { getToken } from "next-auth/jwt";
const token = await getToken({ req, secret: process.env.NEXTAUTH_SECRET });
```

**Two auth tiers in API routes:**
- **Hard auth** (returns 401/403): `/api/search/global`, all `/api/admin/*` routes
- **Soft auth** (token optional, used for logging only): resource CRUD routes (customers, deliveries, clusters, settings, upload)

**Password reset flow**: Forgot password â†’ generates `crypto.randomBytes(32)` token with 1hr expiry â†’ stores in `passwordResetTokens` â†’ **returns token in API response for dev auto-redirect** (no email sending configured).

# API Route Conventions

All routes in `app/api/*/route.ts`. Follow this pattern:

```ts
export async function METHOD(req: NextRequest) {
  try {
    // 1. Optionally extract token: const token = await getToken({ req, secret: process.env.NEXTAUTH_SECRET })
    // 2. Parse body: const body = await req.json()
    // 3. Validate inputs inline (no zod)
    // 4. Execute DB operation
    // 5. Optionally log access: await logServerAccess(req, token)
    // 6. Optionally log activity: await logActivity(...)
    // 7. Return NextResponse.json(...)
  } catch (error) {
    return NextResponse.json({ message: "..." }, { status: 500 });
  }
}
```

**Response conventions:**
- Collections wrapped: `{ customers: [...] }`, `{ deliveries: [...] }`, `{ clusters: [...] }`
- Single resources returned directly (unwrapped)
- Errors: mostly `{ message }`, some use `{ error }` (inconsistency)
- Status codes: 200 (OK), 201 (created), 400 (validation), 404 (not found), 409 (conflict), 403 (forbidden), 500 (server error)
- Dynamic route params: always await: `const resolvedParams = await params; const id = resolvedParams.id`

**Bulk operations**: `/api/customers/bulk`, `/api/deliveries/bulk` accept arrays. Admin bulk delete routes under `/api/admin/*/bulk-delete` use `inArray` with Drizzle.

**File upload** (`/api/upload`): Multipart form via `req.formData()`. Saves to `public/static/uploaded_images/{house_image|proof-of-deliveries}/`. Returns `{ url }`.

**Google Maps URL resolver** (`/api/resolve-maps`): Follows redirects and parses coordinates from multiple URL patterns.

# Logger (`lib/logger.ts`)

Three functions writing to dedicated tables:
- `logActivity(userId, userName, action, details?, targetId?)` â†’ `logs` table
- `logError(userId?, userName?, errorName?, errorMessage?, stackTrace?, pathname?)` â†’ `errorLogs`
- `logAccess(userId?, userName?, pathname, method, ipAddress?, userAgent?)` â†’ `accessLogs`
- `logServerAccess(req, token?)` â€” convenience wrapper for access logging from API routes

All logs silently swallow errors. Write routes log both access + activity.

# Component Architecture

**All components are `"use client"`** and live in `components/`. Pages can be either server or client components.

**Provider hierarchy** (`app/providers.tsx`):
```
SessionProvider â†’ LanguageProvider â†’ ThemeProvider â†’ ToastProvider â†’ ConfirmationProvider â†’ ErrorBoundary
```

**Provider-provided hooks:**
- `useLanguage()` â†’ `{ t(key), locale, setLocale }`
- `useTheme()` â†’ `{ theme, toggleTheme(e?), setTheme(theme, opts?) }`
- `useToast()` â†’ `{ showToast(msg, type?) }` â€” types: `success | error | info | warning`
- `useConfirmation()` â†’ `{ askConfirmation({ title, message, confirmText?, cancelText?, type? }) }` â€” returns Promise<boolean>

**Styling conventions:**
- CSS variables for theme colors: `bg-card`, `bg-background`, `text-primary`, `text-secondary`, `border-card-border`, `bg-surface-hover`
- Defined via Tailwind v4 `@theme inline` in `globals.css`
- Dark mode via `.dark` class + `dark:` prefix on every component
- Font sizes use explicit pixel values: `text-[15px]`, `text-[13px]`, `text-[11px]`, etc.
- All interactive elements use `active:scale-90` for tactile feedback
- Button classes defined in globals.css: `btn-primary`, `btn-secondary`, `btn-outline`, `btn-danger`
- Glassmorphism: `bg-card/80 backdrop-blur-xl`
- Input styling: `inputClass` constant defined locally per component (same pattern copied everywhere)

**Framer Motion patterns:**
- Spring config: `{ type: "spring", stiffness: 400, damping: 25 }` (common)
- `AnimatePresence` for modals, toasts, dropdowns
- `layoutId="nav-indicator"` for shared layout animation in BottomNav
- `motion.div` with `whileTap={{ scale: 0.92 }}` for micro-interactions
- Page transitions in `app/template.tsx`: `{ opacity: 0, y: 15 }` â†’ `{ opacity: 1, y: 0 }`

**Modal patterns:**
- Simple: `{isOpen && (<div className="fixed inset-0 z-50 ...">)}` with backdrop click to close + scroll lock via `document.body.style.overflow`
- AnimatePresence-based: used in ConfirmationProvider, AdminWipeData

**Suspense boundaries** are used in pages that call `useSearchParams()` (Next.js requirement):
- `/deliveries/page.tsx`, `/deliveries/new/page.tsx`, `/customers/page.tsx`, `/clusters/page.tsx`, `/reset-password/page.tsx`

**Two page layout patterns:**

   ```tsx
     <main className="mx-auto max-w-3xl p-4 sm:p-6">...</main>
   ```

2. **Full-height fixed header** (map, admin pages) â€” inner scroll container:
   ```tsx
   <div className="flex flex-col h-[100dvh] bg-background overflow-hidden">
   ```

# Design Quirks & Known Issues

1. **BottomNav auto-hide** does NOT work on pages using inner scroll containers (Pattern 2 above). To fix, switch to body-level scroll or refactor BottomNav to use IntersectionObserver.
2. **coordinate fields** (`latitude`/`longitude`) and `codAmount` are `text` type in DB â€” may need casting when doing numeric operations.
3. **PUT vs PATCH semantics reversed**: `/api/deliveries/[id]` has PUT as partial update and PATCH as full overwrite.
4. **Error response inconsistency**: Some routes use `{ message }`, others use `{ error }` for error responses.
5. **No input validation library** (no zod, no joi) â€” validation is done inline.
6. **No unit tests or integration tests** visible anywhere in the codebase.
7. **Forgot-password** returns token in API response body (dev shortcut â€” remove for production).
8. **Admin sub-pages** for clusters, customers, and deliveries show placeholder content ("under construction").
9. **Soft auth on resource endpoints**: POST/PUT/DELETE on customers, deliveries, clusters proceed even without a valid token (auth check is middleware-only for page routes).
10. **No rate limiting or CSRF protection** implemented.
11. **Severe network performance issue**: Fetching lists of customers, deliveries, or clusters takes ~50s with network spikes exceeding 3.5MB/s. Suspected causes: Supabase connection overhead, no pagination/limit on list queries, or excessive relational includes (e.g., `with: { clusters: { with: { cluster: true } } }` loading N+1 for every row). **Fix later**: add server-side pagination + limit, lazy-load relations, and investigate Supabase pool config. **Workaround for slow connections**: implement optimistic local creation (create record in IndexedDB/local state first, sync to server in background) and add loading skeletons with timeout fallback UI.

# Drizzle ORM Usage Patterns

Three query styles used interchangeably:

1. **Relational query API**: `db.query.customers.findMany({ with: { clusters: { with: { cluster: true } } } })`
2. **Raw SQL**: `sql<number>\`count(...)\`` â€” used for aggregates (cluster customer counts)
3. **Standard query builder**: `db.select()`, `db.insert().values()`, `db.update().set()`, `db.delete().where()`

`db` is exported from `lib/db.ts`: `drizzle(client, { schema })` where schema is the full `./schema` module.
